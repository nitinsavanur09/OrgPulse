Setup order matters
Create services in this exact order: Salesforce Dev Edition first (takes longest to activate — do it first), then Supabase, then ngrok. The Connected App callback URL needs your ngrok URL — but ngrok changes on every restart, so update it fresh each session before sending the auth link to a client.

CREATE TABLE connected_orgs (
  id               UUID PRIMARY KEY DEFAULT gen_random_uuid(),
  salesforce_org_id VARCHAR(18) UNIQUE NOT NULL,
  org_name         TEXT,
  instance_url     TEXT,
  status           TEXT DEFAULT 'connected',

CREATE TABLE scan_runs (
  id               UUID PRIMARY KEY DEFAULT gen_random_uuid(),
  org_id           UUID REFERENCES connected_orgs(id),
  status           TEXT DEFAULT 'pending',
  ai_readiness_index SMALLINT,
  has_hard_blocker BOOLEAN,
  completed_at     TIMESTAMPTZ,
  created_at       TIMESTAMPTZ DEFAULT NOW()
);

CREATE TABLE scan_domain_scores (
  id          UUID PRIMARY KEY DEFAULT gen_random_uuid(),
  scan_run_id UUID REFERENCES scan_runs(id),
  domain      TEXT NOT NULL,
  score       SMALLINT,
  is_blocker  BOOLEAN DEFAULT FALSE
);

CREATE TABLE scan_findings (
  id             UUID PRIMARY KEY DEFAULT gen_random_uuid(),
  scan_run_id    UUID REFERENCES scan_runs(id),
  domain         TEXT,
  severity       TEXT,  -- 'critical' | 'warning' | 'info'
  title          TEXT,
  description    TEXT,
  evidence       TEXT,  -- e.g. "COUNT query: 142,844 / 420,130"
  effort_days    NUMERIC(4,1),
  impact_score   SMALLINT
);

Week 1 — Infra + OAuth
Set up all services, create the project scaffold, and get OAuth working end-to-end against your own Dev Edition org. End goal: running npm run test-conn returns your user count.
Success criterion for Week 1
You click a URL in your browser → Salesforce login screen → you approve → your terminal logs "Connected. Active users: 12" (or whatever your Dev Edition count is). Everything after this is scan logic on top of a working connection.
Week 2 — Scan Tools
Build all 7 domain scan functions. Each one is a self-contained TypeScript function — build and test each against your Dev Edition org before moving to the next. Never return raw data from a scan function.
The zero-copy rule in every scan function
Pattern for every function: (1) run SOQL/API call, (2) extract signal into typed variables, (3) null the raw response immediately, (4) return only the signal. TypeScript enforces this via the return type. If you're returning more than numbers, percentages, counts, and string labels — you're violating the contract.
Week 3 — Scoring Engine
Turn raw signals into domain scores (0–100), detect hard blockers, calculate the weighted AI Readiness Index, and build findings strings. This is your core IP — the rubric encoded as code.

import fs from 'fs'
import path from 'path'
import { ReportData } from './json-schema'

// Load template once at module init
const TEMPLATE = fs.readFileSync(
  path.resolve(__dirname, '../../templates/orgpulse-v2.html'), 'utf-8'
)

export function generateReport(data: ReportData): string {
  // Inject data as window.REPORT_DATA before </head>
  const script = `<script>window.REPORT_DATA=${JSON.stringify(data)}</script>`
  return TEMPLATE.replace('</head>', script + '</head>')
}

Week 5 — Delivery Pipeline
Wire everything together into a single command: connect org → run scans → score → generate report → upload → get signed URL. By end of this week you can run one command and get a client-ready report URL.
scripts/run-scan.ts — the one command that does everythingTypeScript
#!/usr/bin/env tsx
// Usage: npm run scan -- --org [orgId]

import { getConnection } from '../src/auth/connection'
import { runAllScans } from '../src/scan'
import { scoreFindings } from '../src/scoring/engine'
import { buildReportData } from '../src/report/json-schema'
import { generateReport } from '../src/report/generator'
import { uploadReport } from '../src/report/storage'
import { saveResults } from '../src/db/queries'

async function main() {
  const orgId = process.argv[3]
  if (!orgId) throw new Error('Pass --org [orgId]')

  console.log('🔌 Connecting to org...')
  const conn = await getConnection(orgId)

  console.log('🔍 Running scans...')
  const signals = await runAllScans(conn)

  console.log('📊 Scoring domains...')
  const scores = scoreFindings(signals)
  console.log(`   Index: ${scores.overallIndex}/100`)
  console.log(`   Blockers: ${scores.hardBlockers.join(', ') || 'none'}`)

  console.log('📄 Generating report...')
  const reportData = buildReportData(orgId, signals, scores)
  const html = generateReport(reportData)

  console.log('☁️  Uploading to Supabase Storage...')
  const { signedUrl } = await uploadReport(orgId, html)

  await saveResults(orgId, scores, reportData)

  console.log(`\n✅ Done! Report URL (90 days):\n${signedUrl}`)
}

main().catch(console.error)
Week 6 — End-to-End Test
Run the full pipeline against your own Dev Edition org. Fix every rough edge before a client sees it. This week is about hardening, not building.
Do not skip this week
The worst time to discover a bug is during a live client engagement. This week exists specifically to create a safe environment to break things. Run the full pipeline 3+ times. Intentionally test edge cases — what happens if the org has zero Knowledge articles? What if all users are active? What if the Security Health Check returns an unusual score? Every edge case you handle now is a client issue you prevent.
Week 7 — First Pilot Delivery
Your first real client org. This week is delivery, not development. Use the runbook from the pilot delivery guide. Log every step — what took longer than expected, what surprised you, what the client reacted to most.
What success looks like at end of Week 7
A real client has received a signed URL to their V2 OrgPulse report. You have run the readout call. You have sent a follow-up email with a next-step proposal. You have a written feedback form response. And you have a "what I rebuilt manually" log with at least 5 items — this is your Phase 2 product backlog.
Key Decisions
Decisions made in this build plan and why. Read this before you start so you don't accidentally undo them.
🚫
No BullMQ / job queue in this phase
Scans run synchronously in the run-scan.ts script. For a single pilot client, this is fine — scans take 2–4 minutes. Add BullMQ when you're running concurrent scans (client 3+). Adding it now adds complexity with zero benefit.
🚫
No frontend / Next.js in this phase
You run scans via npm run scan from your terminal. Clients receive a signed URL — they never log into a dashboard. A frontend is only needed when clients need self-serve access. That's Phase 2.
🚫
No Claude agent in this phase
The narrative in the report is written by you, manually, in the findings-builder.ts templates. The Claude API is not called at all in this phase. Add the agent after engagement 5 when you know exactly what narrative patterns always appear.
✅
ngrok instead of Render for pilot
For a single pilot, your laptop runs the OAuth callback server. ngrok tunnels the Salesforce redirect to localhost:3000. Zero deployment overhead, zero cost. Switch to Render when you need the server to be always-on for multiple clients.
✅
tsx for development, tsc for production
tsx runs TypeScript directly without a compile step — fast for development and scripts. tsc compiles to dist/ for production deployment on Render.
✅
window.REPORT_DATA injection pattern
The V2 HTML template reads from window.REPORT_DATA — injected as a script tag before </head>. Same template works for sample display (hardcoded fallback data) and live client reports. One template, zero duplication, no templating engine needed.
⚠️
Anthropic DPA — required before paid client
The Claude API is not used in this phase, so this is not blocking. But before you add the agent (Phase 2), you must sign the Anthropic Data Processing Agreement. Contact enterprise@anthropic.com. Do not go live with the agent for EU clients without a signed DPA.
